    **Supabase browser client (apps/web/lib/supabase/client.ts):**
    Follow Pattern 1 from RESEARCH.md exactly:
    ```typescript
    import { createBrowserClient } from '@supabase/ssr'
    export function createClient() {
      return createBrowserClient(
        process.env.NEXT_PUBLIC_SUPABASE_URL!,
        process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY!
      )
    }
    ```

    **Supabase server client (apps/web/lib/supabase/server.ts):**
    Follow Pattern 1 from RESEARCH.md exactly — uses cookies() from next/headers. The try/catch in setAll handles Server Component context where cookies are read-only.

    **Supabase middleware helper (apps/web/lib/supabase/middleware.ts):**
    Extract the Supabase client creation + token refresh logic into a reusable helper called `updateSession(request)`. This creates the Supabase server client within the middleware context, calls `supabase.auth.getUser()` to refresh the token (CRITICAL: use getUser() not getSession() — getSession() does not validate JWT on server), and returns the response with updated cookies.

    **Route middleware (apps/web/middleware.ts):**
    Follow Pattern 2 from RESEARCH.md:
    - Import and call `updateSession(request)` from the helper
    - Get user via `supabase.auth.getUser()`
    - If no user and path starts with `/(app)` or `/dashboard` -> redirect to `/auth`
    - If user exists and path is `/auth` -> redirect to `/dashboard`
    - Matcher: exclude _next/static, _next/image, favicon.ico, api/webhooks (webhooks must not require auth)

    **Auth callback route (apps/web/app/(auth)/auth/callback/route.ts):**
    This handles the redirect back from OAuth and magic link flows:
    - GET handler that reads `code` from searchParams
    - Exchanges code for session via `supabase.auth.exchangeCodeForSession(code)`
    - On success: redirect to `/dashboard`
    - On error: redirect to `/auth?error=auth_callback_failed`
    - CRITICAL: Do NOT use Link components in this route — prefetching can fire before cookie is written (Pitfall 5 from RESEARCH.md). Use programmatic redirect only.

    **Shared auth schemas (packages/shared/src/schemas/auth.ts):**
    Create Zod schemas for cross-platform validation:
    - `loginSchema`: { email: z.string().email(), password: z.string().min(8) }
    - `signupSchema`: { email: z.string().email(), password: z.string().min(8), fullName: z.string().min(2).max(100) }
    - `magicLinkSchema`: { email: z.string().email() }
    Export types inferred from schemas: LoginInput, SignupInput, MagicLinkInput.

    **Update packages/shared/src/index.ts** to re-export from schemas/auth.

    **Mobile Supabase client (apps/mobile/lib/supabase/client.ts):**
    Follow Pattern 6 from RESEARCH.md exactly — uses AsyncStorage, autoRefreshToken, persistSession, detectSessionInUrl: false.
