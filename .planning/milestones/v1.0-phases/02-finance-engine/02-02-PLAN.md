    - Test: Source row has negative amountCents, destination row has positive amountCents
    - Test: Source account balance decremented by amountCents, destination account balance incremented by amountCents
    - Test: Both balance updates use atomic SQL increment (sql`balance_cents + ${delta}`)
    - Test: createTransaction with type='income' inserts one row with positive amount and increments balance
    - Test: createTransaction with type='expense' inserts one row with negative amount and decrements balance
  </behavior>
  <action>
    1. Create `apps/web/lib/finance/queries.ts`:
       - 'use server' (these are called from RSC pages)
       - Import createClient from '@/lib/supabase/server', createDb from '@floow/db', and schema tables
       - Helper: getOrgId() — gets user via supabase.auth.getUser(), extracts orgId from user.app_metadata.org_ids[0], throws if not found
       - getAccounts(orgId: string): select all accounts where orgId matches and isActive = true, ordered by name
       - getTransactions(orgId: string, opts?: { limit?: number, offset?: number, accountId?: string }): select transactions with optional accountId filter, joined with category (for name/color/icon), ordered by date DESC, limit default 50
       - getCategories(orgId: string): select categories where orgId matches OR orgId IS NULL (system defaults), ordered by type then name

    2. Create `apps/web/lib/finance/actions.ts`:
       - 'use server'
       - Import createClient, createDb, schema tables, Zod schemas from @floow/shared, sql and eq from drizzle-orm
       - createAccount(formData: FormData):
         - Get orgId via getOrgId helper (call createClient, extract from JWT)
         - Validate with createAccountSchema.parse({ name: formData.get('name'), type: formData.get('type') })
         - Insert into accounts with orgId, return the account
         - revalidatePath('/accounts')

       - createTransaction(formData: FormData):
         - Get orgId
         - Parse formData: accountId, categoryId, type, amountCents (parse from string), description, date, transferToAccountId
         - Validate with createTransactionSchema
         - Determine signedAmount: income = +amountCents, expense = -amountCents
         - If type === 'transfer':
           - Generate transferGroupId = crypto.randomUUID()
           - Insert TWO transaction rows: debit (source account, negative amount) + credit (destination account, positive amount), both with same transferGroupId
           - Atomic balance update BOTH accounts: source -= amountCents, destination += amountCents
           - Use sql`balance_cents + ${delta}` for atomic increment (NOT read-modify-write)
         - If income/expense:
           - Insert ONE transaction row
           - Atomic balance update: account.balanceCents += signedAmount
         - revalidatePath('/transactions')
         - revalidatePath('/accounts')
         - Return the transaction(s)

    3. Create `apps/web/__tests__/finance/actions.test.ts`:
       - Mock @floow/db (createDb, schema tables), @/lib/supabase/server (createClient), next/cache (revalidatePath)
       - Mock db.insert().values().returning() and db.update().set().where() chains
       - Test createTransaction with type='transfer':
         - Verify db.insert(transactions) called with two rows
         - Verify both rows have same transferGroupId (non-null UUID)
         - Verify source row amountCents is negative, destination row amountCents is positive
         - Verify db.update(accounts) called twice with sql`balance_cents + ${delta}` pattern
       - Test createTransaction with type='income': one insert, positive amount, one balance update
       - Test createTransaction with type='expense': one insert, negative amount, one balance update
       - Follow the mocking pattern from apps/web/__tests__/billing/webhook.test.ts (vi.mock, dynamic import)

    CRITICAL: Use `sql\`balance_cents + ${signedAmount}\`` for atomic balance updates — never read balance, compute, then write back (race condition per Pitfall 5 in research).
    CRITICAL: For transfers, both inserts + both balance updates must succeed together. Wrap in a single db connection's transaction if possible, or at minimum ensure both rows are inserted.
